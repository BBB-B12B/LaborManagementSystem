# After-Sale Q&A: การเปรียบเทียบโครงสร้างและข้อกำหนดระหว่างระบบ After-Sale และ Labor Management (LB)

เอกสารนี้จัดทำขึ้นเพื่อชี้แจงและตอบคำถามเกี่ยวกับความสัมพันธ์ทางโครงสร้างข้อมูลและกฎเกณฑ์ในการทำงานร่วมกันระหว่างระบบ After-Sale และระบบ Labor Management (LB) โดยอ้างอิงตามโค้ดสคริปต์และโครงสร้างข้อมูลบนฐานข้อมูลจริง

---

## 1. ประเด็นเรื่องระดับความละเอียดของงานซ่อม (Task vs Subtask)

> **คำถาม**: ในการกรอกข้อมูลงานซ่อมปกติของแผงโฟร์แมน 1 รายการงานซ่อม (เช่น "ทาสีผนังห้องนอน") จะนับเป็น 1 Task แล้วมี 1 Subtask ผูกคู่กันเสมอ (1-to-1) ใช่หรือไม่? หรือ 1 Task สามารถมีหลาย Subtasks ขนานกันได้? และถ้าเป็น 1-to-1 เราสามารถใช้ default subtaskId เป็นรูปแบบ `[taskId]-0001` ได้เลยหรือไม่?

### คำตอบ:
*   **ไม่ใช่ความสัมพันธ์แบบ 1-to-1 เสมอไป**: โครงสร้างในระบบ LB ได้รับการออกแบบให้ 1 Task สามารถมีหลาย Subtasks ขนานกันได้ (1-to-Many) เพื่อรองรับกรณีที่งานซ่อมหลัก 1 รายการประกอบด้วยงานย่อยหลายจุด เช่น
    *   **Task**: งานซ่อมสีบ้านเลขที่ 99/9
    *   **Subtask 1**: ทาสีผนังห้องนอน
    *   **Subtask 2**: ทาสีฝ้าเพดาน
*   **การคำนวณความก้าวหน้า (Progress)**: เมื่อมีหลาย Subtasks ความก้าวหน้าเฉลี่ยของ Task หลักจะคำนวณจากค่าเฉลี่ยของความก้าวหน้าของ Subtasks ทั้งหมดมารวมกัน
*   **โครงสร้าง Subtask ID**:
    *   ในกรณีที่สร้าง Task ใหม่ที่มีเพียง 1 รายการงานย่อย (1-to-1) หรือสร้าง Subtask แรกของงานนั้น **สามารถใช้รูปแบบ `[taskId]-0001` เป็นค่าเริ่มต้นได้เลย**
    *   ในระบบหลังบ้าน (ใน `TaskService.ts`) เมื่อมีการบันทึกจะใช้รูปแบบเลขรันนิ่ง 4 หลักต่อท้าย `[taskId]` เสมอ:
        ```typescript
        const subtaskNum = (index + 1).toString().padStart(4, '0');
        const subtaskId = `${taskId}-${subtaskNum}`; // เช่น TASK-0001-0001, TASK-0001-0002
        ```

## 2. ประเด็นเรื่องพาธการบันทึกรายงานประจำวัน (Daily Reports Path)

> **คำถาม**: ในระบบ LB จัดเก็บรายงานรายวันไว้ลึกมาก คืออยู่ที่ `subtasks/{subtaskId}/revisions/{revisionId}/dailyReports/{date}` หมายความว่าในการลงรายงานรายวันจริงของช่างหน้างาน ช่างจะป้อนข้อมูลเข้าสู่ Revision ล่าสุดเสมอ (เช่น `rev00` ในการทำงานรอบแรก และหากโดน Reject ตีกลับ จะป้อนเข้าสู่ `rev01` ในรอบแก้ไข) ใช่หรือไม่? และในการดึงประวัติรายงานย้อนหลังเพื่อคำนวณค่าแรง ระบบดึงมาจากทุกๆ revisions/* หรือดึงเฉพาะตัวล่าสุด?

### คำตอบ:
*   **การป้อนข้อมูลประจำวัน**: **ใช่ ช่างหน้างานจะป้อนข้อมูลรายงานประจำวันเข้าสู่ Revision ล่าสุด (Current Revision) ของ Subtask นั้นๆ เสมอ**
    *   **รอบแรก**: ระบบกำหนดเป็น `rev00` รายงานจะเขียนลงที่ `revisions/rev00/dailyReports/{date}`
    *   **เมื่อมีการตีกลับ (Reject)**: ระบบจะสร้าง Revision ถัดไปโดยเพิ่มเลขรันนิ่ง (เช่น `rev01`, `rev02`) และรีเซ็ต `dailyProgress` บนตัว Subtask เป็น 0 รวมถึงปรับสถานะเป็น `upcoming` เพื่อให้ช่างเข้ามาลงรายงานรอบแก้ในพาธ `revisions/rev01/dailyReports/{date}`
    *   *ข้อจำกัดย้อนหลัง*: ระบบมีกฎควบคุมเพื่อป้องกันไม่ให้ช่างลงงานย้อนหลังก่อนวันที่โดน Reject (Revision Timeline Boundary Validation)
*   **การดึงข้อมูลเพื่อคำนวณค่าแรง**: **ระบบดึงจากทุกๆ `revisions/*` และ `help/*` (Support) ที่อยู่ในงวดค่าแรงนั้นทั้งหมด**
    *   การคำนวณค่าแรง (ใน `WagePeriodService.ts`) จะใช้การดึงข้อมูลด้วยคำสั่ง `collectionGroup('dailyReports')` เพื่อกวาดรายงานประจำวันทั้งหมดในโครงการตามช่วงวันที่ของงวดค่าแรง (StartDate ถึง EndDate) โดยไม่แยกว่าอยู่ภายใต้ Revision ใด
    *   **เหตุผลเชิงธุรกิจ**: เพื่อให้ช่าง/แรงงานได้รับเงินครบตามวันที่ปฏิบัติงานจริง แม้ว่าผลงานในรอบแรก (`rev00`) จะโดนปฏิเสธ (Reject) และต้องแก้ไขงานในรอบถัดไป (`rev01`) ก็ตาม แรงงานที่เข้าปฏิบัติงานในทุกรอบจะยังคงได้รับค่าจ้างตามวันเวลาที่ทำงานจริง

## 3. ประเด็นเรื่องชื่อฟิลด์ผู้รับผิดชอบหลัก (Assignees Schema)

> **คำถาม**: ในระบบ After Sale เดิมเก็บผู้รับผิดชอบเป็นรหัสพนักงานง่ายๆ ในฟิลด์ `responsibleStaffIds: string[]` แต่ในระบบ LB ใช้เป็น `assignees: Array<{ employeeId, name, roleId }>` อยากทราบว่าฟิลด์ `roleId` ในอาร์เรย์ `assignees` มีค่าใดบ้างที่ยอมรับ (เช่น "Foreman", "Technician")? และจำเป็นต้องกรอกให้ครบทั้ง 3 ฟิลด์ย่อยเสมอเมื่อมีการมอบหมายงานหรือไม่?

### คำตอบ:
*   **ค่าที่ยอมรับใน `roleId`**:
    ระบบ LB กำหนดให้ผู้รับผิดชอบดูแลงานซ่อมเป็นพนักงาน/เจ้าหน้าที่ของบริษัท (ไม่ใช่แรงงานชั่วคราว) โดยยอมรับค่าบทบาทหน้าที่ (Role Code) ภาษาอังกฤษ 2 ตัวตามโครงสร้างการตรวจสอบสิทธิ์และการจัดการของระบบ ดังนี้:
    *   `AM` : Admin / Area Manager (ผู้ดูแลระบบ / ผู้จัดการพื้นที่)
    *   `FM` : Foreman (โฟร์แมนควบคุมงานหน้างาน)
    - `SE` : Site Engineer (วิศวกรสนาม)
    - `OE` : Operations Executive / Office Engineer (ฝ่ายปฏิบัติการ)
    - `PE` : Project Engineer (วิศวกรโครงการ)
    - `PM` : Project Manager (ผู้จัดการโครงการ)
    - `PD` : Project Director (ผู้อำนวยการโครงการ)
    - `MD` : Managing Director (กรรมการผู้จัดการ)
*   **ความจำเป็นในการกรอกข้อมูล**: **จำเป็นต้องระบุให้ครบถ้วนทั้ง 3 ฟิลด์ย่อยเสมอ** (`employeeId`, `name`, และ `roleId`)
    *   เนื่องจากใน Interface `TaskAssignee` ฝั่งหลังบ้านได้รับการกำหนดให้เป็นฟิลด์บังคับทั้งหมด (ไม่มีเครื่องหมาย `?` เป็น Optional):
        ```typescript
        export interface TaskAssignee {
          employeeId: string;
          name: string;
          roleId: string;
        }
        ```
    *   การกรอกไม่ครบจะส่งผลให้ระบบการกรองสิทธิ์แสดงผลงานบนหน้าแผงควบคุมของโฟร์แมน (FM) และวิศวกรสนาม (SE) ผิดพลาด หรือทำให้การประมวลผลล้มเหลวได้

## 4. ประเด็นเรื่องการจับคู่สถานะ (Status Mapping)

> **คำถาม**: เพื่อป้องกันโลจิกการแสดงสีและสถานะหน้าบ้านพัง ขอถามการจับคู่สถานะ (Status Mapping) ระหว่างสองระบบว่าตรงกันตามนี้หรือไม่:
> *   'Pending' / 'Assigned' (ของเรา) → 'upcoming' (ของ LB)
> *   'In Progress' (ของเรา) → 'in-progress' (ของ LB)
> *   'Completed' (ของเรา) → 'for-checking' (ของ LB - รอตรวจรับ)
> *   'Verified' / 'Approved' (ของเรา) → 'approved' (ของ LB - สำเร็จปิดใบงาน)
> *   'Rejected' (ของเรา) → 'rejected' (ของ LB - ส่งคืนแก้ไข)

### คำตอบ:
การจับคู่ตรงกันเกือบทั้งหมด แต่มี**ข้อแตกต่างสำคัญ**ในทางปฏิบัติของระบบ LB ดังนี้:

| สถานะ After-Sale | สถานะในระบบ LB | ผลลัพธ์และโลจิกการทำงานในระบบ LB |
| :--- | :--- | :--- |
| **'Pending' / 'Assigned'** | `upcoming` | **ถูกต้อง** (เมื่อมีการสร้างงานซ่อมย่อยใหม่ สถานะเริ่มต้นจะเป็น `upcoming`) |
| **'In Progress'** | `in-progress` | **ถูกต้อง** (ระบบจะเปลี่ยนสถานะเป็น `in-progress` อัตโนมัติเมื่อช่างลงบันทึกความก้าวหน้าแรกเข้ามา) |
| **'Completed'** | `for-checking` | **ถูกต้อง** (หมายถึงงานเสร็จสิ้น 100% จากหน้างานและรอผู้ควบคุมงานมาตรวจรับ) |
| **'Verified' / 'Approved'** | `completed` | **ไม่ใช่ `approved` แต่เป็น `completed`**<br>ใน Model ของระบบ LB ไม่มีสถานะชื่อ `'approved'` โดยเมื่อกดอนุมัติงานในระบบ LB (`POST /api/tasks/:id/approve`) ระบบจะเปลี่ยนสถานะ Task เป็น `completed` เพื่อปิดใบงานถาวร |
| **'Rejected'** | `upcoming` (ใน Revision ใหม่) | **ระบบ LB จะไม่มีสถานะชื่อ `'rejected'` ค้างไว้**<br>เมื่อมีการกดตีกลับงาน (`POST /api/tasks/:id/reject`) หลังบ้านของระบบ LB จะสร้างรอบการแก้ไขใหม่ (Revision ใหม่) แล้วทำการปรับสถานะของ Task และ Subtasks คืนกลับไปเป็น `upcoming` และรีเซ็ตความก้าวหน้าเป็น 0 เพื่อเริ่มเก็บข้อมูลรอบใหม่ทันที |

> [!NOTE]
> รูปแบบสถานะทั้งหมดที่กำหนดไว้ในประเภทข้อมูล `TaskStatus` ของระบบ LB ได้แก่:
> `export type TaskStatus = 'upcoming' | 'in-progress' | 'rework' | 'for-checking' | 'completed';`
